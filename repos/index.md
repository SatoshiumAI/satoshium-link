# Satoshium Repo Map

This page groups the major Satoshium repositories by architectural role.

It is not a complete inventory of every repository in the workspace. Instead, it provides a public map of the repositories that most directly shape the platform’s knowledge layer, trust layer, intelligence layer, simulations, and documentation surface.

---

## How to Read the Repo Map

The repo map mirrors the broader platform structure.

Some repositories define stable foundations. Some support reasoning and intelligence workflows. Others are experimental, internal, or public-facing.

Together they form the working repository ecosystem behind Satoshium.

**Categories used in this map:**

- Foundational repositories
- Active build repositories
- Experimental repositories
- Supporting / internal repositories

---

## Platform Alignment

The repo map reflects the layered structure of the platform:

- **Architecture** explains the layered model of the platform
- **Systems** documents functional software systems
- **Build** explains engineering direction over time
- **Registry** provides canonical public identifiers for system-level components

---

## Repo Map Purpose

This map helps:

- show where the most important repos fit in the architecture
- separate stable foundations from experiments and public interfaces
- clarify how public pages relate to underlying repositories
- support long-term traceability as the ecosystem grows

---

## Canon & Specifications Layer

- **satoshium-canon** — constitutional truth layer for platform definitions and claims
- **satoshium-specs** — structured specifications for agents, models, lifecycle logic, and standards
- **satoshium-governance** — governance doctrine and coordination concepts informing future rules

These repositories move slowly and define the conceptual structure inherited by later systems.

---

## Knowledge Engine & Documentation Layer

- **satoshium-glossary** — canonical terminology base
- **glossary-builder** — interactive terminology interface
- **satoshium-docs** — structured documentation and references
- **satoshium-journal** — development record and milestone logs
- **satoshium-master-timeline** — long-horizon sequencing of platform work

Shared language and documented history are part of the platform itself.

---

## Trust, Safety & Verification Layer

- **satoshium-trust** — trust primitives and verification logic
- **satoshium-safety** — safety doctrine and protective design patterns
- **satoshium-aegis** — agent firewall direction
- **satoshium-certifier** — readiness and certification logic
- **satoshium-registry** — structured indexing concepts for platform components

Trust and verification are architectural requirements, not optional add-ons.

---

## Intelligence, Core Logic & Agent Layer

- **satoshium-core** — shared platform logic and reusable utilities
- **satoshium-interface** — interface direction for interacting with systems
- **satoshium-agent-lab** — sandbox for testing agent architectures
- **satoshium-agents** — future aligned agent systems layer
- **satoshium-curator** — canon-aware curation and structured updates
- **satoshium-echobot / satoshium-saynobot** — applied behavioral experiments

Agents emerge from the knowledge, trust, and governance layers rather than replacing them.

---

## Simulation, Labs & Scenario Layer

- **satoshium-labs** — experimental public exploration layer
- **satoshium-scenarios** — structured scenario modules
- **satoshium-simulations** — model-driven testing environments
- **satoshium-demos** — smaller public prototypes
- **satoshium-future-uses** — applied scenario thinking for long-horizon use cases

Simulation converts abstract ideas into testable structures.

---

## Signal & Coordination Layer

- **satoshium-signal** — infrastructure signal routing surface
- **signal-boards** — structured intake interface for coordination signals
- **sovereign-signal** — scenario-driven signaling environment

The Signal Layer connects proposals, simulations, governance surfaces, and future service interactions before they become formal systems.

---

## Public Surface & Experience Layer

- **satoshium-progress** — public build log and architecture gateway
- **satoshium-site** — public narrative platform surface
- **satoshium-ui** — visual system alignment work
- **satoshium-hub** — potential coordination entry layer
- **satoshium-academy** — future structured onboarding surface

These repositories shape how the platform is encountered publicly.

---

## Supporting Infrastructure & Internal Utility Repos

- **satoshium-data** — structured data inputs and assets
- **satoshium-devtools** — internal developer workflows
- **satoshium-utils** — shared utilities
- **satoshium-internal** — coordination notes and internal materials
- **satoshium-drafts** — staging space for early documents
- **satoshium-template** — structural baseline for aligned repositories

Not every repo is a public system. Some exist to support workspace coherence.

---

## Media, Storytelling & Narrative Layer

- **satoshium-chronicles** — narrative framing and long-form history
- **satoshium-glyphs** — symbolic visual language
- **satoshium-founder / satoshium-mirror** — reflective and founder-facing spaces

Satoshium is engineered, but also interpreted and explained over time.

---

## Reading the Workspace as a Whole

The full Satoshium workspace is larger than the public repo map shown here.

This map highlights repositories most directly supporting:

- the constitutional layer
- the Knowledge Engine
- trust systems
- intelligence systems
- simulation environments
- public documentation surfaces

As the ecosystem grows, system-critical repositories should remain aligned with the Architecture page, Build page, and Registry so the platform remains understandable rather than merely expansive.
